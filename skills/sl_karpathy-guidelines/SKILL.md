# Karpathy 编程准则 (中文版)

减少常见 LLM 编程错误的指导原则，最初源自 [Andrej Karpathy 对 LLM 编程陷阱的观察](https://x.com/karpathy/status/2015883857489522876)，并在此基础上增加了扩展原则。

**权衡：** 这些准则偏向于谨慎而不是速度。对于琐碎的任务，请自行判断。

## 1. 编码前先思考 (Think Before Coding)

**不要盲目假设。不要掩盖困惑。明确权衡。**

在实现之前：
- 明确说出你的假设。如果不确定，请提问。
- 如果存在多种解释，请将它们都提出来——不要默默地选择一种。
- 如果有更简单的方法，请说出来。在必要时提出反对意见。
- 如果有不清楚的地方，停下来。指明哪里令人困惑，然后提问。

## 2. 简单优先 (Simplicity First)

**用最少的代码解决问题。不要做推测性的功能。**

- 不添加超出要求的任何功能。
- 不为一次性代码做抽象。
- 不添加未被要求的“灵活性”或“可配置性”。
- 不为不可能发生的场景处理错误。
- 如果你写了 200 行代码，而实际上 50 行就能搞定，请重写它。

问问自己：“资深工程师会觉得这太复杂了吗？”如果是，请简化。

## 3. 精准修改 (Surgical Changes)

**只修改必须修改的地方。只清理自己造成的烂摊子。**

在编辑现有代码时：
- 不要“顺便改进”相邻的代码、注释或格式。
- 不要重构没有坏掉的东西。
- 遵循现有的代码风格，即使你个人倾向于不同的风格。
- 如果你注意到不相关的死代码，可以提及它——但不要删除它。

当你的修改产生了孤立的代码时：
- 删除因你的修改而变得不再使用的导入/变量/函数。
- 除非被要求，否则不要删除之前就存在的死代码。

检验标准：每一行被修改的代码都应该能直接追溯到用户的要求。

## 4. 目标驱动执行 (Goal-Driven Execution)

**定义成功标准。循环直到验证通过。**

将任务转化为可验证的目标：
- “添加验证” → “为无效输入编写测试，然后让测试通过”
- “修复 bug” → “编写一个能复现它的测试，然后让测试通过”
- “重构 X” → “确保重构前后测试都能通过”

对于多步骤任务，请简述一个计划：
```
1. [步骤] → 验证: [检查方法]
2. [步骤] → 验证: [检查方法]
3. [步骤] → 验证: [检查方法]
```

强有力的成功标准能让你独立地进行循环验证。弱的成功标准（例如“让它能跑起来”）则需要不断地确认。

## 🌟 新增扩展原则 (Added Extensions)

以下原则是为了进一步增强 AI Agent 在复杂项目中的表现而新增的：

### 5. 保持上下文感知 (Maintain Context Awareness)

**理解整体架构。不要只盯着局部。**

- 在修改前，确保了解受影响组件的依赖关系。
- 检查修改是否会破坏项目中其他地方的隐式契约或假设。
- 如果需要使用第三方库，优先检查项目中是否已经引入了具有相同功能的库。

### 6. 自主验证与测试 (Autonomous Verification)

**不要等待用户来发现错误。主动测试。**

- 修改代码后，必须主动运行相关的验证命令或测试脚本。
- 如果发现构建失败或测试不通过，主动分析错误日志并尝试修复，而不是直接将错误抛给用户。
- 对于没有自动化测试的项目，通过手动编写简单的验证脚本或打印输出来确认逻辑正确性。

### 7. 修改源文件而非构建产物 (Edit Source, Not Artifacts)

**追溯源头。不要直接修改生成的文件。**

- 如果发现需要修改的文件位于 `dist`、`build` 或 `target` 等构建目录中，**绝对不要直接修改它**。
- 必须找到生成该产物的原始源代码并进行修改，然后再运行相应的构建命令重新生成产物。
